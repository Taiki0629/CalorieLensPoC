# 10 開発プロセス規約

方針: **AIレビューは厚く／人間への質問は最小限**。夜間に自律で回り、翌朝には進んでいる状態を目指す。
人間が止めるのは **課金一括実行** の直前のみ（`/CLAUDE.md` §2）。決定背景は `decisions/ADR-0001`。

自動連結（オーケストレーション）の仕様は `50_自動化ワークフロー規約.md`。本書は各工程の中身とレビュー観点を定める。

---

## 0. モデル振り分け（Fable / Opus）

| 用途 | モデル | 例 |
|------|--------|-----|
| Jira起票（ドラフト・書き込み）、人間への質問・確認 | **Fable** | 課題文面の生成、`createJiraIssue` |
| 起草(設計)・各種レビュー・実装・テスト（品質を要する作業） | **Opus** | 設計ドキュメント、design-reviewer、コード実装、動作確認 |

サブエージェントは frontmatter の `model` で固定。Workflow では段ごとに `model` を指定する。

---

## 1. フロー① Jira起票

```
要件 → 課題ドラフト(Fable) → ticket-reviewer でレビュー(Opus) → 修正 → Jira書き込み(Fable)
```

1. **ドラフト化(Fable)**: `20_Jira運用規約.md`＋`templates/Jira課題テンプレート.md` に沿い、要件を課題（タイプ・サマリ・背景/目的・AC・DoD・関連）に落とす。まだ書き込まない。
2. **レビュー(Opus)**: `ticket-reviewer` サブエージェントで点検。ACが機械判定可能か、階層・スコープ・推測混入などを見る。「要修正」ならドラフトに戻す（最大2回）。
3. **書き込み(Fable)**: 「起票可」で `createJiraIssue`。エピック→ストーリーの順に、親を指定して作成。

スキル: `ticket`（`/ticket <要件>` で起動）。一括はWorkflow `intake`。

---

## 2. フロー② 開発（1チケット＝1サイクル）

```
要件確認 → 起草(20_design,Opus) → 設計レビュー(Opus) → 実装(Opus)
 → コードレビュー(Opus) → テスト/動作確認(Opus) → 記録 → クローズ
```

| # | 工程 | モデル | やること | 完了条件 |
|---|------|--------|----------|-----------|
| ① | 要件確認 | Opus | 対象Jira課題のAC/DoDを読む。曖昧なら整える | ACが機械判定可能 |
| ② | 起草 | Opus | 設計/方針を **`20_design/<課題キー>-*.md`** に draft で書く | ACを満たす設計 |
| ③ | 設計レビュー | Opus | `design-reviewer` で点検 → 反映。**承認は自動**（重大ゼロで確定） | 重大指摘ゼロ |
| ④ | 実装 | Opus | コードを書く。課題キーをブランチ・コミットへ | ACを満たし動く |
| ⑤ | コードレビュー | Opus | `/code-review` ＋ 実験系は `experiment-validator` → 修正 | 指摘対応・lint通過 |
| ⑥ | テスト/動作確認 | Opus | 小さく動かして実装できているか確認。手順・結果を `40_test/` に記録。**課金一括は人間承認** | 動作確認記録あり |
| ⑦ | 記録 | Opus | 介入ログを1段落（`templates/介入記録テンプレート.md`） | 追記済み |
| ⑧ | クローズ | Opus | commit → push → PR作成 → マージ → Jira を Done（**すべて自律**） | マージ・Done |

## 3. レビュー観点（多観点・すべてOpus）

「良いと思います」で終わらせない。観点ごとにチェックリストで合否を出す。

- **設計** → `design-reviewer`: 目的整合／AC網羅／**推測検出**／再現性／スコープ／記事化。
- **コード** → `/code-review`（汎用）＋ `experiment-validator`（実験特化: **数値手打ち禁止**／ログ完全性／コスト算出／堅牢性／再現性／秘密情報）。
- **記事** → `article-reviewer`: 数値の出所／出典／注意書き／過度な一般化。
- **起票** → `ticket-reviewer`: AC機械判定／DoD網羅／階層／スコープ／推測。

自動承認の原則: **重大指摘ゼロなら次工程へ自動で進む**。重大が残る場合のみ修正ループ（最大2回）→ なお解決しなければ人間へエスカレーション。

---

## 4. 承認ゲート（人間が止める唯一点）

| ゲート | タイミング | ふるまい |
|--------|-----------|----------|
| **課金一括実行** | 工程⑥ | 想定リクエスト数・概算コストを提示しGOを待つ。承認後に本番実行 |
| Jira 削除/大量変更 | 随時 | 実行前に確認（単発の作成/更新/遷移/コメントは自動） |
| 外部公開 | 記事完成後 | Qiita投稿は人間 |

設計確定・commit/push・PR作成・マージは **自律**（承認不要）。

---

## 5. 記録の原則（記事②のため）

- **決定**は ADR（`decisions/`）に「何を・なぜ・代替案・結果」で残す。
- **AIに任せた範囲／人の介入**をチケットごとに介入ログへ1段落（スキル `intervention-log`）。
- 採用率の体感・介入回数を貯め、記事②「どこまで任せられたか」を数字で語れるようにする。
