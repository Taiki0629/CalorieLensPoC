# ADR-0001: ルール体系とプロセス設計の制定

- ステータス: 採用
- 日付: 2026-06-30
- 関連課題: （プロジェクト作成後に紐づけ）

## 背景 / 文脈

料理写真カロリー推定 PoC を、Jira 起点の AI 駆動開発で進める。成果物は Qiita 記事2本（テーマ本体＋AI駆動開発の方法論）。記事②があるため「進め方の記録が残ること」自体が成果物となる。開始時点で CLAUDE.md・Claude Code 設定・Jira 連携・各規約は未整備だった。

## 決定

1. **ルールを4階層で構成**: ①ルート `CLAUDE.md`（憲法）②`10_management/` の各規約（プロセス/Jira/Git/記事）③`30_development/CLAUDE.md`（実装規約・次バッチ）④横断（決定ログ・テンプレ・`.claude/` 設定）。
2. **Jira 連携は「作成・更新まで自動」**: 検索・作成・説明/AC更新・ステータス遷移・コメント・リンクは Claude が自動。削除・大量一括変更のみ人間承認。
3. **プロセスは重量級**: 課題単位ブランチ＋PR必須、複数サブエージェントによる多観点レビュー、承認ゲート多め。記事②の素材を厚くするため。
4. **技術スタックは Python + uv + ruff**。
5. **不変の原則**: 言語=日本語（コード識別子は英語）／推測禁止（モデルID・料金・vision対応・API挙動は実確認）／実験数値の手打ち禁止（ログから生成）／秘密情報は `.env` のみ。
6. **記録の強制**: 決定は ADR、各サイクルで介入ログを1段落残す。

## 代替案

- **Jira 連携を「提案のみ」**: 安全だが手数が増え、自動化の知見（記事②の核）が薄くなるため不採用。
- **軽量プロセス（main直・チャット内レビュー）**: 速いが工程の記録が薄く、記事②の素材にならないため不採用。
- **venv+pip**: 枯れているが、uv の方が再現性・速度で優位なため不採用。

## 結果 / 影響

- 規約が明文化され、AI も人も同じ基準で判断・レビューできる。
- 重量級ゆえ手数・トークン消費は増えるが、それ自体が記事②の観察対象になる。
- 残課題: Jira プロジェクト `CAL` の手動作成、レビュー自動化（`.claude/agents`・`commands`）、`30_development/CLAUDE.md` の整備（次バッチ）。
