# 料理写真カロリー推定 PoC 設計書
## OpenAI GPT vs ai& Inference — 費用・性能ガチ比較

> Qiita投稿キャンペーン「OpenAI・Claude APIと互換性あり！国内AIモデル推論プラットフォーム『ai& Inference』を使ってみよう」向け PoC

---

## 1. このPoCで何をやるのか（一言で）

料理写真を見せて「カロリー・料理名・PFC」をAIに推定させ、**OpenAIの最新GPTモデル**と**ai& Inferenceの激安オープンモデル**を、**費用**と**性能**の両面で比較する。

さらに「**写真の枚数・アングルを増やすと精度はどこで頭打ちになるか**」を実験し、AIに料理を撮らせるときの費用対効果の最適点を探る。

### 検証したい問い

- 料理写真からのカロリー推定は、どのモデルがどれくらい正確か
- アングルを変えた写真を足していくと、精度は何枚目で頭打ちになるか
- OpenAIの数十分の一のコストのオープンモデルは、どこまで戦えるか
- 食事写真という個人の健康データを扱うとき、国内データセンターで完結できる構成にどんな意味があるか

## 5. 実験① アングル積み増し実験（本PoCの主役）

「写真を増やすと精度は上がるのか、何枚目で頭打ちになるのか」を検証する。

### 撮影プロトコル

同じ料理を、**角度を半分ずつ刻みながら**撮影し、1枚ずつ積み増していく。

| ステップ | 使用する画像（累積） | 枚数 |
|---------|---------------------|------|
| S1 | 真上（0°） | 1枚 |
| S2 | 真上 + 斜め45° | 2枚 |
| S3 | 真上 + 45° + 22.5° | 3枚 |
| S4 | 真上 + 45° + 22.5° + 11.25° | 4枚 |

> ※ 角度は「真上＝0°」を基準に、45° → 22.5° → 11.25° と半分ずつ刻む。頭打ちが早く来る場合はS3〜S4で打ち切ってよい。
> ※ 複数枚は、1リクエストの `content` 配列に複数の画像を入れて「同じ料理を多面的に見せる」形で投げる。

### 見たいこと

- 枚数を増やすと誤差（後述APE）は単調に下がるのか、途中で頭打ちか、むしろ悪化するか
- 枚数増加に伴う**コスト増**（画像はトークン換算で課金）と、精度改善の**割に合うポイント**はどこか
- モデルごとに「頭打ちの枚数」は違うのか

## 6. 実験条件（その他の決定事項）

### 入力は「写真のみ」

料理名や分量のヒントは与えず、**写真だけ**から判断させる。AIの素の実力を測る。

### 出力はカロリーだけでなく構造化情報も

まぐれ当たり（料理名を外したのにkcalが偶然近い）を見抜くため、料理名なども出力させ、独立した軸で採点する。

**出力フォーマット（JSON固定）**

```json
{
  "dish_name": "料理名",
  "total_kcal": 数値,
  "protein_g": 数値,
  "fat_g": 数値,
  "carb_g": 数値,
  "confidence": 0〜1の数値
}
```

### 試行回数

各条件を `temperature=0` でも **3回ずつ**実行し、出力のブレ（分散）も記録する。「同じ写真・同じ設定でも、このモデルは±◯◯kcalブレる」という再現性の知見を得る。

### リクエスト総数の目安

```
2品 × 4ステップ(S1〜S4) × 5モデル × 3試行 = 120 リクエスト
```

## 7. 評価軸

### A. 性能 — カロリー精度

正解値との **絶対パーセント誤差（APE）** で採点。小さいほど優秀。

```
APE = |推定kcal − 正解kcal| / 正解kcal × 100
```

3試行の平均と分散（ブレ幅）の両方を見る。

### B. 性能 — 料理名の同定精度

料理名を「正解 / 惜しい / 誤り」で手動ラベリングし、**カロリー精度とは独立した軸**として集計。
→「名前は当てるがカロリー過大」「名前は外すが量感は近い」といったモデルの性格の違いを描く。

### C. 性能 — 速度

各リクエストの latency（応答時間）を計測。安いモデルが速いとは限らないため独立軸とする。

### D. 費用 — 1リクエストあたり実コスト

レスポンスの `usage`（入力/出力トークン）× 各モデル単価で実額を算出。画像枚数が増えるS2〜S4でコストがどう跳ねるかも見る。

---

## 8. 費用の見せ方 — 「1日あたり」で逆算する

絶対額（1リクエスト◯円）はピンと来ないので、**実務スケールに逆算**して提示する。

> 例：「毎日1万人がカロリー撮影するアプリを運用したら、**1日あたり** GPT-5.4 なら ◯◯円、Qwen3.6 なら ◯円」

- 試算式: `1日あたりコスト = 1リクエストあたり実コスト × 想定リクエスト数/日`
- 想定シナリオは「DAU 1万人 × 1人1回撮影 = 10,000リクエスト/日」を基準に置く
- ai& の激安さ（Qwen3.6 が ¥0）がこの逆算で最大に際立つ

---

## 9. 考察軸 — 国内データセンターの意味

カロリーアプリ自体は機密ではないが、**食事写真は個人の健康データ**であることに触れる。

> 食事記録・体型・既往症と結びつく食事写真を海外クラウドに送ることに抵抗があるユーザーや、医療・ヘルスケア事業者にとって、**国内データセンター × オープンモデル**でカロリー推定を完結できる構成は、クローズドな海外APIでは満たせない要件に応えられる。

「互換APIで手軽」「多彩なモデルで安い」に加えて、この一文で3つ目の強みも回収する。

## 10. 実装ステップ

| ステップ | 内容 |
|---------|------|
| **S0. 準備** | `/models` で ai& 側の vision 対応モデルIDを確定。すき家・コンビニ弁当の正解kcalを確定 |
| **S1. 撮影** | 2品をそれぞれ 0° / 45° / 22.5° / 11.25° の4アングルで撮影 |
| **S2. 実行** | base_url を切り替えながら全モデル・全条件を3試行実行。usage と latency を記録 |
| **S3. 採点** | APE・料理名ラベリング・速度・コストを集計 |
| **S4. 可視化** | モデル別ランキング、枚数×精度の折れ線、コスト×精度の散布図、1日あたりコスト試算表 |
| **S5. 執筆** | 結果と考察（頭打ちの枚数 / 激安モデルの健闘 / 国内DCの意義）をまとめて投稿 |

### 中核コードのイメージ（OpenAI互換・base_url差し替え）

```python
from openai import OpenAI

# ai& も OpenAI も同じSDK。base_url を変えるだけ
aiand = OpenAI(base_url="https://api.aiand.com/v1", api_key="sk-aiand-...")
openai = OpenAI(api_key="sk-openai-...")  # 本家

def estimate(client, model, image_b64_list):
    content = [{"type": "text", "text": PROMPT}]
    for b64 in image_b64_list:  # 複数アングルを1リクエストに
        content.append({
            "type": "image_url",
            "image_url": {"url": f"data:image/jpeg;base64,{b64}"}
        })
    resp = client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": content}],
        temperature=0,
    )
    return resp  # .choices / .usage / latency を回収
```

## 11. 記事の構成（伸びる順序）

1. **掴み** — 「牛丼とコンビニ弁当のカロリー、AIは当てられる？しかも国産DCの激安モデルで」
2. **手軽さ** — base_url 1行差し替えで本家もオープンモデルも同じコードで叩ける
3. **実験設定** — 2品・アングル積み増し・写真のみ・JSON出力
4. **結果①** — 枚数 × 精度の折れ線（何枚目で頭打ちか）
5. **結果②** — モデル別 精度ランキング（カロリー誤差・料理名正解率）
6. **結果③** — 費用：1日あたりコスト逆算表
7. **考察** — 激安モデルの健闘度 / アングルの費用対効果 / 国内DCの意義
8. **締め** — 健康データ × 国産オープンモデルという選択肢

### タイトル案

> 牛丼とコンビニ弁当のカロリーをAIに当てさせたら？ 最新GPT-5.4 vs 国産激安オープンモデルを費用と精度でガチ比較
