# Claude Code に PoC を丸ごと任せてみた ― Jira 起点の「重量級 AI 駆動開発」でどこまで自律できたか

料理写真からカロリーを推定するモデル比較 PoC（[記事①](#)）を、**要件の起票から実装・レビュー・
実験実行・記事執筆まで**、できる限り AI（Claude Code）に任せて開発しました。本記事は、その
**進め方そのもの**を題材にします。「どこまで AI に任せられ、どこで・なぜ人が介入したか」を、
実際のログと具体例で振り返ります。

> 本記事の数値・事例はすべてプロジェクトの `介入ログ` と Git/Jira 履歴に基づく実記録です。

## 1. 全体像 ― なぜ「重量級」にしたのか

軽量に「AIにコードを書かせる」だけなら簡単です。しかし今回は**あえて重量級**にしました。

- すべての作業を **Jira 課題**に紐づける（ブランチ名・コミット・設計書に課題キー `CAL-12` を含める）。
- 1課題 = 1サイクルで、**設計 → 設計レビュー → 実装 → コードレビュー → テスト → 記録 → クローズ
  （commit/push/PR/マージ/Jira更新）**を回す。
- モデルを役割で振り分け（起票・質問は軽量モデル、設計・レビュー・実装・テストは高性能モデル）。

理由は2つ。①**あとで記事にできる記録**を残すこと自体が成果物だから。②AIに広く任せるほど、
「歯止め（ゲートとルール）」を先に設計しておかないと事故る、という仮説を検証したかったからです。

## 2. 憲法（CLAUDE.md）でルール化したこと

プロジェクト直下に `CLAUDE.md` を「憲法」として置き、人間もAIも従う最上位ルールにしました。
効いたのは次の4つです。

| ルール | 中身 | 狙い |
|--------|------|------|
| **承認ゲート** | 課金が発生するAPIの一括実行・外部公開（記事投稿）・Jira の削除/大量一括変更だけは人間承認。それ以外は自律 | 「止める点」を最小に絞り、夜間でも自律で走らせる |
| **推測の禁止** | モデルID・料金・vision対応・APIの挙動は推測で書かない。実API（`/models`等）と公式情報で確定し**確認日と出典**を残す | 「それっぽいが間違い」を構造的に排除 |
| **数値の手打ち禁止** | 実験の表・グラフ・コスト試算は必ずスクリプトがログ（JSONL）から生成。記事に数値を直接打たない | 転記ミスと恣意的な数字を根絶 |
| **秘密情報** | APIキーは `.env` のみ・コミット厳禁。ログにも残さない | 事故を仕組みで防ぐ |

このうち **承認ゲートを「課金一括・外部公開・Jira の削除/大量一括変更」だけに絞った**のが肝でした。
本PoC全体で、AIの自律を止めた**課金一括の停止点はわずか2回**（弁当48件・牛丼12件の実験実行）だけです
（1〜数件の少額な疎通確認は自律で実施）。起票・設計確定・実装・レビュー・commit・PR・マージ・Jira更新は、
すべて人間の承認なしで回りました。

## 3. Jira × Claude（MCP）で自動化できたこと／できなかったこと

Atlassian の MCP 連携で、Claude が Jira を直接読み書きします。

**自動化できた**（人間承認なしで完走）:
- 課題の取得・検索、**ステータス遷移**（未着手→進行中→レビュー中→完了）、コメント追記
- 設計ドキュメントの起草・確定、実装、テスト、**Git のブランチ作成・commit・push・PR作成・マージ**
- 課題ごとの受け入れ条件(AC)/DoD に沿った自己チェックとクローズ

**できなかった／人間が必要だった**:
- **課金一括実行の GO**（実験本番）＝設計上の停止点
- **外部データ**：正解カロリー（弁当のパッケージ表示、すき家公式サイト）、撮影した料理写真
- **外部公開**：Qiita への投稿（この記事も未投稿・承認待ち）

つまり自動化の壁は「技術」ではなく「**お金・物理・公開**」という、そもそも人間が責任を持つべき境界に
きれいに一致しました。ここは自動化しない方が良い、という設計判断そのものが成果です。

## 4. 多観点 AI レビューが実際に捕まえた欠陥

本プロジェクトで最も価値を出したのが、**専門特化のサブエージェントによる多観点レビュー**です。
「設計レビュー」「実験の妥当性検証」「記事レビュー」を役割ごとに分け、確定・マージ・投稿の前に必ず通しました。
実際に投稿・マージ前に捕まえた例（多くは人間が見落としやすい種類）:

- 🐛 **コスト会計のバイアス**（実験検証レビュー）：JSON リトライ時に usage が最後の1回で上書きされ、
  **安いモデルほどコストが系統的に過少計上**されるバグ。本PoCの中核主張（費用比較）を直接歪めるもので、
  全試行の usage 合算に修正。
- 🐛 **パス解決バグ**（コードレビュー）：相対パスでの設定読み込み時に出力先が別ディレクトリに化け、
  図が規約と違う場所に出ていた（実際に踏んでいた）。回帰テスト付きで修正。
- 🐛 **証跡と記録の食い違い**（設計レビュー）：ドキュメントが「モデルはこう答えた」と書いているのに、
  保存された生ログの値と違っていた（プローブを2回実行し2回目が証跡を上書き→未保存の結果を引用していた）。
  恒久証跡へ転記し「記録＝証跡」に統一。
- 🐛 **憲法との矛盾**（設計レビュー）：実験ステップを勝手に拡張した設計が、用語集・コスト基準と矛盾。差し戻し。
- ✍️ **誇張と事実誤認**（記事レビュー）：「単価が数十分の一」は誇張（実測は十数分の一）、
  「国産オープンモデル」は事実誤認（比較したのは Google/Moonshot 製で、国産なのは推論データセンターの方）。
  投稿前に訂正。

多くは「動くけれど間違っている」タイプで、単体テストや目視では見つけにくいものです（最後の誇張・
事実誤認は文言問題ですが、読者が調べれば気づく類でした）。
**AIに広く書かせるほど、AIに厳しくレビューさせる**――この非対称な組み合わせが効きました。

## 5. 「推測の禁止」が実害を防いだ2つの瞬間

ルールが絵に描いた餅で終わらず、実際に**誤りを止めた**場面です。

1. **激安モデルの目玉が消えた**：企画当初は「¥0 の Qwen3.6 が激安の主役」の想定でした。しかし
   実API（`/models` の capability）を叩くと **Qwen は画像入力に非対応**。推測で書いていたら
   「使えないモデル」を主役に据えるところでした。→ vision 対応を実証済みの Gemma に差し替え。
2. **正解カロリーが14%ずれるところだった**：すき家 牛丼の正解を検索すると「795 kcal」。しかし
   公式PDFをメニュー名と値を厳密に対応させて読むと、795は「高菜明太マヨ牛丼」の値で、**素の牛丼は695 kcal**。
   検索値を鵜呑みにしていたら、実験の分母（正解値）が14%ずれ、全結論が狂っていました。

「出典と確認日を必ず残す」という一手間が、**PoCの土台そのもの**を守りました。

## 6. どこまで任せ、どこで人が介入したか（まとめ）

- **コードはほぼ全量 AI 生成**。人間はコードを書いていません。設計の初稿採用率は体感7〜8割で、
  残りはレビュー指摘に沿って AI 自身が修正しました。
- **人間の入力は「意思決定・外部データ・承認」に集約**：品数の方針、料理写真、正解カロリー、
  実験実行のGO、記事の方向づけ。総じて**数十文字の指示**が中心でした。
- **AIの自律を止めた課金一括のゲートは全体で2回**、実験の総課金は**約¥50**（実ログ集計 64件で¥48.7＋
  疎通の数円）。それ以外（起票〜マージ〜Jira更新〜
  記事ドラフト）は人手の承認なしで流れました。

「AIに任せる／人が握る」の線引きは、**責任の所在（お金・公開・一次情報）**で引くのが自然でした。

## 7. うまくいったこと・ハマったこと・次にやるなら

**うまくいった**
- 承認ゲートを最小化しつつ、多観点レビューと推測禁止で品質を担保できた。
- 「設定の外出し（config集約）＋数値手打ち禁止」で、記事の数値・図をログから機械生成。
  最終出力では転記事故ゼロ（過程で起きた集計の取り違えは公開前に自己検出・訂正）。
- キーや正解値が未到着でも、**合成データの二経路**を先に作って採点・可視化を完成させ、待ち時間を消した。

**ハマった**
- テストが実環境（`.env`）に依存し、一度**本物のAPIを呼んでしまった**（幸い 404/401 のエラーで弾かれ
  課金は発生せず。ただし有効な呼び出しなら課金し得た）＝課金リスクの露呈。テストを密閉化して再発防止。
- 使い捨てプローブを2回実行し、2回目が証跡を上書き→設計ドキュメントが「未保存の結果」を引用。証跡の永続化で解決。
- 「集計は overall／per-dish」など、**数字の出所を取り違える**ミス。手打ち禁止ルールが「出所を毎回問う」習慣として機能し、自己検証で捕捉。

**次にやるなら**
- サブエージェントを段ごとに自動連結する Workflow は、自動マージの安全性を検証してから解禁したい。
- 「証跡＝記録」を最初からコミット対象に置く（gitignore 下に一次証跡を置かない）。

## 8. 締め ― 再現できる「AI駆動開発のテンプレ」

今回の学びを、そのまま真似できる形にすると：

1. **憲法を1枚書く**（`CLAUDE.md`）。最上位ルールと、止める点（承認ゲート）だけを最小限に定義。
2. **止める点は「お金（課金一括）・外部公開・Jira の削除/大量一括変更」に絞る**。それ以外は自律させる。
   一次情報（正解値・写真など）は"ゲート"ではなく"人間しか出せない入力"として区別する。
3. **推測を禁止し、出典と確認日を強制**する。実APIと公式情報で確定する。
4. **数値は手打ち禁止**。設定を外出しし、表・図はログから生成する。
5. **AIに広く書かせ、別のAIに厳しくレビューさせる**（多観点・特化エージェント）。確定・公開の前に必ず通す。
6. **1課題1サイクルで記録を残す**（介入ログ）。これがそのまま次の記事の素材になる。

AIは「広く速く書く」のは得意ですが、**間違え方も自信満々**です。だからこそ、書かせる範囲を広げるほど、
**歯止め（ルール）と第三者チェック（多観点レビュー）を先に設計する**ことが、実用的なAI駆動開発の鍵でした。
